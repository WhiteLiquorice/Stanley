Privacy Policy — Stanley Browser Butler

Last updated: June 2025

Stanley Browser Butler ("Stanley", "the extension", "we") is a Chrome extension that provides local browser automation. This policy explains what data the extension collects, how it is used, and how it is protected.

1. Data Collected and Stored Locally

Stanley stores the following data on your device using chrome.storage.local. This data never leaves your device except as described in Section 2.

  • Authentication tokens — your email address, user ID, and a Firebase ID token, used to keep you signed in to Stanley's AI services.
  • Workflow history — a log of up to 20 recent workflow prompts and their outcomes (completed or failed). Used to display your history in the extension popup.
  • Saved workflows — prompt templates you choose to save for reuse.
  • Preferences — settings such as Active Mode (headful vs headless automation).

2. Data Sent to External Services

Certain features require communication with external cloud services. The following types of data may leave your device:

  • Workflow prompt text — when you submit a workflow in plain English, the text of your prompt is sent to Firebase Cloud Functions to be interpreted by an AI model and converted into automation steps. Your Firebase ID token is included for authentication.
  • Screenshots — when the automation engine cannot locate a UI element by text or label, it captures a JPEG screenshot of the current page and sends it to Firebase Cloud Functions for visual element resolution and layout analysis. Screenshots are used only to resolve element locations and are not stored permanently by the server.
  • DOM element metadata — element names, IDs, ARIA labels, and placeholder text are sent to help AI models match user descriptions to page elements. This data is not stored permanently.

Future cloud services: Stanley may add new cloud-based features (advanced data extraction, multi-page workflow optimization, analytics, or collaboration). All new features will require explicit user opt-in, be authenticated via your Firebase account, return only data payloads (never executable code), and be documented in this policy before release.

No browsing history, tab URLs, full page content, or personal information is sent to external services beyond what is described above.

3. Technical Details: Cloud Services Architecture

Why Cloud-Based?

Stanley's AI features (workflow compilation and visual element resolution) rely on large language models and vision models that are computationally expensive and impractical to run locally:

  • Model size: Modern AI models range from 7–70+ GB. Bundling these with the extension would make it unusable.
  • Inference hardware: Language and vision models require GPU or TPU for acceptable latency. Users should not be forced to buy expensive hardware just to use browser automation.
  • Battery & performance: Local inference would drain battery, consume RAM, and slow down your computer for every workflow.

What Gets Sent?

  • Workflow prompts: Plain text only (e.g., "log into Mindbody and extract client names"). No credentials, passwords, or sensitive data.
  • Screenshots: Low-quality JPEG (55% compression, ~50KB). The browser UI, address bar, and other tabs are excluded — only the active page is captured.
  • Element metadata: Names, IDs, ARIA labels, and placeholder text of interactive elements. The full page DOM and content are never sent.

What Comes Back?

Only structured data — never executable code or binary files:

  • Action arrays: [{ action: "click", description: "..." }, ...]
  • Element coordinates: { x: 100, y: 200, element: "button" }
  • Localization strategies: { strategy: "role", value: "Login" }

Security Measures

  • Authentication: Every request includes your Firebase ID token. The server validates this before processing.
  • Encryption: All communication is over HTTPS.
  • No code execution: The extension does not evaluate, parse, or execute returned data as code. The manifest does not grant dynamic script-loading permissions.
  • Server logging: Requests are logged for debugging and abuse detection, then deleted after ~30 days per Google Cloud policies. You can request immediate deletion of your logs via support.

Your Control

You can:

  • Inspect network traffic in Chrome DevTools to see exactly what is sent.
  • Review the Cloud Function source code — it's in the same Firebase project.
  • Disable specific features if you prefer local-only automation.
  • Request that your data be purged from server logs.

3. Native Messaging (Local Daemon)

Stanley communicates with a locally installed companion application (the Stanley daemon) via Chrome's Native Messaging API. This communication is strictly between the Chrome extension and a process running on your own machine. No data passes through a remote server as part of this channel.

4. Data Sharing

We do not sell, rent, or share your personal data with third parties. Data sent to the Firebase Cloud Function is used solely to provide automation planning and element resolution features and is not shared, logged permanently, or used for training.

5. Data Retention

Locally stored data (history, preferences, tokens) remains on your device until you sign out or uninstall the extension. You can clear saved workflows and history from within the extension popup at any time.

6. Security

Authentication tokens are stored in chrome.storage.local, which is scoped to the extension and inaccessible to web pages or other extensions. All communication with Firebase Cloud Functions is performed over HTTPS.

7. Children's Privacy

Stanley is not directed at children under 13. We do not knowingly collect data from children.

8. Changes to This Policy

We may update this policy as features change. The "Last updated" date at the top will reflect any changes. Continued use of the extension after an update constitutes acceptance of the revised policy.
