Bridgeway Suite: Stripe Billing vs Custom Subscription Portals for Clinics

Bridgeway Suite does encapsulate payment orchestration logic within monorepo workspaces to solve Stripe Membership Billing latency issues and Med Spa Front-Desk Operations inconsistency. By leveraging Firebase Cloud Functions alongside React applications, the suite ensures that subscription state remains consistent across distributed microservices while eliminating manual reconciliation overhead during patient scheduling.

Payment Infrastructure Architecture: Server-Side vs Client-Side State Management

The decision between integrating Stripe directly versus building a custom subscription portal hinges on data sovereignty and webhook reliability. In the bridgeway-v2 monorepo structure, payment logic resides in the functions workspace rather than the client-side React apps exposed in apps/*. This separation prevents race conditions where frontend state updates fail to reflect server-side Stripe charges.

Handling Membership Billing via Firebase Functions

Stripe Membership Billing relies on server-side validation to prevent fraud and ensure accurate prorations. Bridgeway replaces generic Node.js payment handlers with Firebase Cloud Functions triggered by Stripe webhooks (stripe.webhook event listeners). This approach allows you to parse Stripe events securely without exposing secrets in the client bundle.

// apps/Mobile/src/services/billing.ts
import { getAuth } from "firebase/auth";
import { stripeWebhookHandler } from "@functions/stripe-handler";

export async function processMembershipCharge(auth: Auth) {
  const session = await getStripeSession(auth.uid);
  
  // Server-side validation ensures the user is active before processing
  if (!session.isActive()) {
    throw new Error("Patient eligibility failed");
  }
}

By moving billing logic to functions, you avoid client-side rendering delays. When a patient updates their subscription tier, the custom portal must handle complex prorations and tax calculations that are error-prone in browser-based JS alone. The backend handles the arithmetic, returning status codes for immediate UI feedback.

Custom Subscription Workflows in React Workspaces

When building a custom portal within apps/*, you manage your own database schema instead of relying on Stripe’s checkout page. This grants full control over patient data retention but requires rigorous testing to ensure billing parity with Stripe’s built-in logic. Use Playwright to verify the checkout flow matches the backend expectations defined in the monorepo configuration.

// apps/Mobile/src/components/SubscriptionPortal.tsx
import { useState, useEffect } from "react";

export function SubscriptionPortal({ patientId }) {
  const [status, setStatus] = useState("pending"); // 'active', 'canceled'

  // Direct state management for custom logic
  useEffect(() => {
    fetch(`/api/subscription/${patientId}`)
      .then(res => res.json())
      .then(data => setStatus(data.status));
  }, [patientId]);
}

Custom portals are necessary when clinics require specific branding or need to bypass Stripe’s checkout page for sensitive medical records. However, you must implement strict validation on the frontend to prevent invalid transactions from reaching your functions workspace.

Operational Efficiency in Med Spa Front-Desk Operations

Medical clinics face unique challenges regarding patient flow and billing synchronization. Bridgeway addresses Patient Scheduling Optimization by decoupling calendar logic from payment processing. When a front-desk agent schedules an appointment, the system must confirm availability without locking up the database indefinitely.

Booking Fee Optimization Logic

Booking fees are distinct from subscription costs but often share the same gateway configuration in package.json. To optimize this, configure Stripe Pricing Plans within your Firebase Cloud Functions to apply different tax rates for membership tiers versus one-off services.

// functions/config/payments.json
{
  "products": [
    {
      "id": "prod_subscription",
      "type": "recurring",
      "currency": "usd"
    },
    {
      "id": "prod_service",
      "type": "one-off",
      "currency": "usd"
    }
  ]
}

This configuration allows you to calculate net revenue before applying fees. The booking_fee logic resides in the database schema, not the checkout UI, ensuring that the front-desk operations team sees accurate financial data for reporting purposes.

Patient Scheduling Optimization with Playwright

To ensure the subscription portal works correctly across different devices, use Playwright to automate end-to-end testing of the scheduling flow. This validates that a confirmed booking triggers the correct Stripe webhook event and updates the Firestore database accordingly.

// apps/Mobile/tests/booking.spec.ts
import { test } from "@playwright/test";

test.describe("Med Spa Front-Desk Operations", () => {
  test("verify subscription activation on schedule confirmation", async ({ page }) => {
    await page.goto("/appointment/schedule");
    
    // Simulate booking action
    await page.click(".submit-payment-button");
    
    // Verify backend state via API request or assertion
    const result = await fetch("/api/subscription/status");
    expect(result.status).toBe("201");
  });
});

This testing strategy catches integration bugs that manual QA misses. When the functions workspace returns a success code, Playwright confirms the UI reflects this change immediately. This reduces front-desk errors and ensures patient data remains consistent across the clinic ecosystem.
